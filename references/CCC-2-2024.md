## Purpose

Establishes the minimum cybersecurity requirements for cloud computing in KSA, applying to both organisations that **operate** cloud services (Cloud Service Providers, CSPs) and organisations that **consume** cloud services (Cloud Service Tenants, CSTs). Developed following a mapping study against international standards including US FedRAMP, Singapore MTCS SS, Germany C5, Cloud Controls Matrix (CCM), and ISO/IEC 27001.

---

## Two-sided model — CSP and CST

Every CCC control is attributed to the Provider role (P), the Tenant role (T), or both. Control identifiers explicitly encode the role: `1-3-P-1-1` is a Provider control; `1-3-T-1-1` is the corresponding Tenant control.

**Always confirm which role the user is assessing before applying controls.**

| Role | Definition | Scope |
|------|-----------|-------|
| **Cloud Service Provider (CSP)** | Any natural or legal person that provides cloud computing services to the public, directly or indirectly, through data centres inside or outside KSA, and manages them in whole or in part | IaaS, PaaS, SaaS providers |
| **Cloud Service Tenant (CST)** | Any natural or legal person that subscribes to cloud computing services from a CSP | Government agencies, CNI entities, and any in-scope entity consuming cloud |

> A single organisation may be both CSP and CST — for example, a government entity providing a shared cloud platform while also consuming a third-party SaaS. Both role sets must be assessed.

---

## Applicability (CCC Scope of Work)

**CSPs in scope:** Any CSP providing cloud computing services to CSTs within scope.

**CSTs in scope:**
- Any government agency in KSA — inside or outside the Kingdom — including ministries, authorities, establishments, and their companies and sub-entities
- All private-sector entities **owning, operating, or hosting Critical National Infrastructures (CNIs)** that currently use or are planning to use any cloud service

**CSPs outside scope (examples):**
- CSPs providing services exclusively to non-Saudi entities outside the Kingdom and not to in-scope CSTs
- CSPs providing services only to individuals or private-sector entities that do not own, operate, or host CNIs, and do not serve in-scope CSTs

NCA strongly encourages all other KSA entities to adopt the controls as best practice.

### Control identification notation

`[Main Domain]-[Subdomain]-[P or T]-[Main Control]-[Subcontrol]`

- **P** = Cloud Service Provider control
- **T** = Cloud Service Tenant control
- Green-coloured numbers in the official document (e.g. 1-3-2) are cross-references to ECC subdomains or controls

Example: **2-2-P-1-3** = Main Domain 2, Subdomain 2, Provider control, Main Control 1, Subcontrol 3.

### 1-1 Cybersecurity Roles and Responsibilities

**Objective:** Ensure roles and responsibilities are defined for all parties involved in cloud cybersecurity controls, including the Authorizing Official (head of the CSP or CST, or their delegate).

| Control Ref | Role | Obligation | ECC parent |
|-------------|------|-----------|------------|
| 1-1-P-1-1 | CSP | Identify, document, and approve cybersecurity roles and RACI assignment for all cloud services stakeholders, including the Authorizing Official's roles | Extends ECC §1-4-1 |
| 1-1-T-1-1 | CST | Identify, document, and approve cybersecurity roles and RACI assignment for all cloud services stakeholders, including the Authorizing Official's roles | Extends ECC §1-4-1 |

**Applicability by level:** Both 1-1-P-1 and 1-1-T-1 are **mandatory at all four levels** (L1–L4).

### 2-2 Identity and Access Management

**Objective:** Ensure secure and restricted logical access to information and technology assets; prevent unauthorised access; allow only authorised access necessary for assigned tasks.

**CSP controls (2-2-P-1 series — extends ECC §2-2-3):**

| Control Ref | Obligation |
|-------------|-----------|
| 2-2-P-1-1 | Manage generic account credentials through an identity and access process — accountability cannot be assigned to a specific individual for such credentials |
| 2-2-P-1-2 | Implement secure session management: session authenticity, session lockout, and session timeout termination |
| 2-2-P-1-3 | Enforce MFA for privileged users and personnel with access to the Cloud Technology Stack |
| 2-2-P-1-4 | Implement a formal process to detect and prevent unauthorised access (e.g. unsuccessful login attempt threshold) |
| 2-2-P-1-5 | Use secure methods and algorithms (e.g. secure hashing functions) for saving and processing passwords |
| 2-2-P-1-6 | Secure management of third-party personnel accounts |
| 2-2-P-1-7 | Enforce access control to management systems and administrative consoles |
| 2-2-P-1-8 | Mask displayed authentication inputs (especially passwords) to prevent shoulder surfing |
| 2-2-P-1-9 | Obtain CST approval before the CSP or CSP's third parties access any CST-related asset *(Optional at Level 4)* |
| 2-2-P-1-10 | Provide capability to immediately interrupt a remote access session and prevent any future access for a user *(Optional at Level 4)* |
| 2-2-P-1-11 | Provide CSTs with MFA services for privileged cloud users |
| 2-2-P-1-12 | Ensure restricted and controlled access to storage systems and means (e.g. Storage Area Network — SAN) |

**CST controls (2-2-T-1 series — extends ECC §2-2-3):**

| Control Ref | Obligation |
|-------------|-----------|
| 2-2-T-1-1 | Manage identity and access for all cloud credentials throughout their full lifecycle |
| 2-2-T-1-2 | Maintain confidentiality of cloud user identification, credentials, and access rights — require all users (employees, third parties, CST personnel) to keep credentials private |
| 2-2-T-1-3 | Implement secure session management on cloud: session authenticity, lockout, and timeout termination |
| 2-2-T-1-4 | Enforce MFA for privileged cloud users |
| 2-2-T-1-5 | Implement formal process to detect and prevent unauthorised cloud access (e.g. unsuccessful login attempt threshold) |

**Applicability:** 2-2-P-1 and 2-2-T-1 **mandatory at all four levels** (with 2-2-P-1-9 and 2-2-P-1-10 optional at Level 4).

### 2-3 Information System and Information Processing Facilities Protection

**CSP controls (2-3-P-1 series — extends ECC §2-3-3):**

| Control Ref | Obligation |
|-------------|-----------|
| 2-3-P-1-1 | Ensure all configurations are applied in accordance with CSP's cybersecurity standards |
| 2-3-P-1-2 | Ensure separation and isolation of data, environments, and information systems across CSTs to prevent data commingling |
| 2-3-P-1-3 | Adopt cybersecurity principles for technical system configurations adhering to the minimum functionality principle |
| 2-3-P-1-4 | Ensure Cloud Technology Stacks can securely handle input validation, exceptions, and failure *(Optional at Level 3 and Level 4)* |
| 2-3-P-1-5 | Fully isolate security functions and applications from other functions and applications in the Cloud Technology Stack |
| 2-3-P-1-6 | Notify CSTs of cybersecurity requirements provided by the CSP that are usable by the CST |
| 2-3-P-1-7 | Detect and prevent unauthorised changes to software programs and systems |

| 2-3-P-1-9 | Community cloud services provided to in-scope CSTs (government and CNI entities) must be isolated from cloud computing provided to entities outside scope *(Not applicable at Level 4)* |
| 2-3-P-1-10 | Deploy modern technologies such as EDR to ensure CSP information servers and devices are ready for rapid incident response |

## Cloud Cybersecurity Controls Levels

CCC-2:2024 defines four classification levels using a top-down approach. Controls are assigned mandatory or optional (recommended) status per level. The **highest applicable level** must be adopted when data of different classification levels is processed together.

| Level | Data classification | Description |
|-------|--------------------|-----------:|
| **Level 1** | Top Secret | Highest classification — most controls mandatory; strictest requirements |
| **Level 2** | Secret | Secret data — nearly all controls mandatory |
| **Level 3** | Confidential | Confidential data — most controls mandatory; some optional |
| **Level 4** | Public | Public data — reduced mandatory set; several controls optional |

## Audit hot spots — triage list

Ranked by frequency in NCA CCC gap assessments. Each item cites the source control.

1. **ECC prerequisite not in place (all domains):** CCC compliance cannot be achieved without ECC baseline. Many entities attempt CCC assessment while ECC §2-2-3 (IAM), §2-12 (monitoring), and §4-2 (cloud baseline) remain non-compliant.
2. **No cloud security RACI / roles document (1-1-P-1-1, 1-1-T-1-1):** Authorizing Official's cloud security responsibilities, and stakeholder RACI across cloud operations, are frequently undocumented.
3. **Cybersecurity positions in KSA data centres not filled with Saudi nationals (1-4-P-1-1):** CSPs with offshore or contractor arrangements in KSA data centre cybersecurity roles are non-compliant.
4. **No cloud-specific risk register or acceptable risk levels (1-2-P-1-1, 1-2-T-1-1):** Entities consume cloud services without a separate cloud risk register or formally approved acceptable risk thresholds.
5. **MFA absent or incomplete for Cloud Technology Stack access (2-2-P-1-3, 2-2-T-1-4):** Privileged access to cloud management planes (console access, admin APIs, hypervisor management) without MFA is a critical gap.
6. **Tenant isolation not tested or contractually confirmed (2-3-P-1-2):** Architecture diagrams may assert isolation but no penetration test or independent attestation exists.
7. **No DDoS protection on cloud-hosted services (2-4-P-1-3):** Cloud-hosted public-facing services without DDoS scrubbing or equivalent protection.
8. **Vulnerability scanning cadence below minimum (2-9-P-1-1):** External CTS components must be scanned at least monthly; internal CTS components quarterly. Many CSPs run quarterly across the board or on an ad-hoc basis only.
9. **Penetration testing does not cover Cloud Technology Stack (2-10-P-1-1):** Pen test scope limited to corporate network; cloud management plane, tenant APIs, and IaaS/PaaS layers excluded.
10. **SIEM not covering full Cloud Technology Stack (2-11-P-1-5):** Log collection missing cloud management plane logs, virtualisation layer logs, or serverless/container orchestration logs.
11. **No real-time incident notification mechanism to CST (2-12-P-1-6):** Contracts may promise notification but no technical SLA or verified communication channel for real-time reporting exists.
12. **Exit strategy / data deletion confirmation absent (2-6-P-1-3, 2-6-T-1-1):** No contractual commitment from CSP for secure data destruction on contract termination; no tested export/migration procedure maintained by CST.
13. **Cryptographic standards non-compliant with NCS-1:2020 advanced level (2-7-P-1-1, 2-7-T-1-1):** Legacy TLS versions in use; non-compliant cipher suites; no HSM for key storage at higher classification levels.
14. **No key management procedures documented (2-15-P-1, 2-15-T-1):** Encryption keys in use but no key ownership, rotation schedule, or backup/recovery mechanism documented.
15. **Community cloud isolation not verified by CST (2-3-T-1-1):** Government and CNI tenants have not verified — by contract or testing evidence — that community cloud services are isolated from entities outside scope.

## Common pitfalls

- **ECC as optional prerequisite:** CCC is explicitly an extension to ECC — ECC gaps directly become CCC gaps. Jumping to CCC without ECC baseline in place is an audit-fail on the opening question.
- **Missing CSP role:** Organisations that operate any internal shared cloud platform are CSPs under CCC, not only CSTs. Both role sets must be assessed.
- **Data localisation assumed resolved:** Removal of in-Kingdom hosting controls from CCC does not eliminate the obligation. NDMO Data Management Standards and sector-specific rules still apply and must be checked.
- **Community cloud isolation treated as CSP-only:** Control 2-3-T-1-1 places an explicit obligation on government and CNI CSTs to *verify* isolation — this requires contractual assurance and testing evidence, not just the CSP's assurance.
- **Vulnerability scanning cadence misread:** The monthly/quarterly cadence in 2-9-P-1-1 applies differently to external versus internal CTS components. Running quarterly across the board for a CSP is non-compliant for external components.
- **Key management in CSP hands only:** For Levels 1–2 (Top Secret / Secret), tenant-managed or externally-held keys are effectively required. Relying solely on CSP-managed keys for high-classification data is a significant gap.
- **No exit strategy:** CSTs frequently have no tested data export or migration procedure. NCA auditors specifically request exit strategy documentation under 2-6-T-1-1.
- **Third-party sub-processor blind spots:** CSPs often have sub-processors (e.g. underlying infrastructure providers) without security assessments, documentation requirements, or disclosure to CSTs — non-compliant under 4-1-P-1-2 and 4-1-P-1-4.
- **SaaS excluded from CCC scope:** Many organisations exclude SaaS applications on the basis that security is the vendor's responsibility. SaaS consumption is CST-role cloud use and applicable CST controls (2-1-T, 2-2-T, 2-6-T, 2-9-T, 2-11-T, 2-15-T, 3-1-T) all apply.
- **Real-time incident notification treated as aspirational:** Control 2-12-P-1-6 requires real-time reporting — this must be a tested, contractually committed mechanism, not a best-effort promise.

## Mapping to CST CCRF

CST regulates cloud licensing (who can operate cloud services in KSA and under what commercial terms); NCA CCC regulates the cybersecurity controls those services must implement. Both apply to CSPs operating in KSA.

| Aspect | CST CCRF (RS10) | NCA CCC-2:2024 |
|--------|-----------------|----------------|
| Cloud service licensing | ✔ | ✗ |
| Cybersecurity controls for cloud services | ✗ | ✔ |
| Data localisation obligations | ✔ (licensing layer) | Transferred to NDMO |
| Customer data protection | Partially (consumer protection) | ✔ (security controls) |
| Service disruption reporting | To CST regulator | To NCA (cybersecurity incidents) |

Holding a CST CCRF licence does not exempt a CSP from NCA CCC obligations. Both must be maintained concurrently.

## Key definitions (from CCC-2:2024 Annex B)

| Term | Definition |
|------|-----------|
| **Cloud Service Provider (CSP)** | Any natural or legal person that provides cloud computing services to the public, directly or indirectly through data centres (inside or outside KSA), and manages them in whole or in part |
| **Cloud Service Tenant (CST) / Cloud Customer** | Any natural or legal person that subscribes to cloud computing services from a CSP |
| **Cloud Technology Stack (CTS)** | Layered architecture of technologies essential for cloud computing services: data centre infrastructure, LAN, storage/compute/hyper-convergence hardware, hypervisor, cloud management platform, virtual appliances, OSs, application software, O&M platforms, cloud security technologies, etc. |
| **Authorizing Official** | Head of the CSP or CST, or their designated delegate, with authority over cybersecurity roles, responsibilities, and approvals |
| **Cloud Computing** | On-demand network access to a shared pool of configurable computing resources (networks, servers, storage, applications, services) that can be rapidly provisioned with minimal management effort. Five essential characteristics: on-demand self-service, broad network access, resource pooling, rapid elasticity, measured service. Service models: SaaS, PaaS, IaaS. Deployment models: private, community, public, hybrid. |
| **Community Cloud** | Cloud deployment shared exclusively among a defined group of organisations (e.g. government entities and CNI entities), isolated from public cloud customers |
| **CMDB** | Configuration Management DataBase — database storing configuration records of systems throughout their lifecycle |
| **RACI Matrix** | Responsible, Accountable, Consulted, Informed Matrix — maps each stakeholder to their degree of involvement in a process or function |
| **SIEM** | Security Information and Event Management — system managing and analysing security event logs in real time for threat monitoring, event correlation, and incident response |
| **NCS-1:2020** | National Cryptographic Standards issued by NCA — the "advanced level" of this standard is the required cryptographic baseline under CCC-2:2024 |
| **Critical National Infrastructure (CNI)** | Assets whose loss or vulnerability may result in significant negative impact on availability of basic services, national security, national defence, or the state economy |
| **EDR** | Endpoint Detection and Response — technology for detecting and responding to endpoint-level threats |
