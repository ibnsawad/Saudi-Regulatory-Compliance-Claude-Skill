# NCA CCC-2:2024 — Cloud Cybersecurity Controls

## At a glance

| Field | Value |
|-------|-------|
| Issuer | National Cybersecurity Authority (NCA), KSA |
| Document | Cloud Cybersecurity Controls (CCC-2:2024) |
| Supersedes | CCC-1:2020 |
| Status | Mandatory for in-scope entities |
| Relationship to ECC | CCC is layered on top of ECC-2:2024 — ECC remains the baseline |

## Two-sided model: CSP and CST

CCC-2:2024 splits obligations into two role sets:

- **Cloud Service Provider (CSP)** — operates the cloud platform.
- **Cloud Service Tenant (CST)** — consumes the cloud service. Often the user's own organisation.

Every control is attributed to one or both roles. Always confirm which role the user is assessing before applying controls.

## Cloud computing levels (criticality tiers)

CCC uses tiering to scale obligations. The general logic (confirm exact level numbering against the published document):

- **Level 1** — lowest sensitivity / non-critical workloads.
- **Level 2 / 3 / 4** — increasing sensitivity, with Level 4 typically reserved for critical national systems and top-secret data.

Tiering drives:

- Permitted deployment models (public / private / hybrid / community)
- Data residency expectations
- Required cryptographic strength and key custody
- Logging, monitoring, and incident-response intensity
- Permissibility of foreign-operated CSPs

Higher-tier workloads typically require in-Kingdom data residency, sovereign key control, and stricter access controls.

## Domain structure

CCC-2:2024 follows the ECC four-main-domain structure, applied to cloud. The four main domains:

1. **Cybersecurity Governance (cloud)** — cloud strategy, policy, roles, cloud-specific risk management, cloud third-party governance.
2. **Cybersecurity Defence (cloud)** — IAM and federation, virtualization and tenant isolation, network and edge protection, key management and HSM, secure configuration of PaaS / IaaS / SaaS, encryption, secure software lifecycle for cloud-native, container and Kubernetes security, logging and monitoring with cloud-native services, cloud incident management, vulnerability management of cloud assets, API security, image and registry hygiene.
3. **Cybersecurity Resilience (cloud)** — cloud business continuity, multi-region / multi-AZ design, backup architecture, recovery testing.
4. **Third-Party Cybersecurity (cloud-context)** — sub-CSP / sub-processor governance, cloud broker arrangements, transit between CSPs.

## Audit hot spots

Most-cited gaps in CCC reviews:

- Misconfigurations exposing storage buckets, databases, or admin consoles
- IAM with excessive permissions (over-privileged human and machine identities)
- Lack of CSP-side and CST-side responsibility matrix in contracts
- Encryption keys held by the CSP rather than the tenant where the tier requires sovereign control
- Logs not exported or retained outside the CSP, leading to evidence gaps on CSP outage
- No CSPM / CNAPP and no defined alerting baseline
- Container / Kubernetes admission controls and image signing missing
- Inadequate tenant-isolation testing in multi-tenant SaaS
- No documented exit / portability plan
- DDoS, WAF, and bot protections absent at internet edge

## Mapping to ECC

Where a CCC control restates an ECC obligation in cloud terms, treat the ECC obligation as primary and the CCC control as the cloud-specific implementation. In assessment output, cite both (e.g. "ECC-2:2024 §X-Y-Z aligned to CCC-2:2024 §A-B-C").

## Mapping to CST CCRF

CST regulates cloud licensing and customer protection; NCA CCC regulates cybersecurity controls. A CSP operating in KSA must comply with both. A CST CCRF licence does not exempt the CSP from NCA controls.

## Required artefacts (CSP)

- Cloud governance policy and roles
- CSP–CST shared-responsibility matrix
- Cloud-specific risk register and treatment plan
- Architecture diagrams covering data flows, key management, and tenant isolation
- Hardening baselines per cloud service in use
- IAM model with privileged-access controls and break-glass procedure
- Continuous-monitoring baseline (CSPM, workload protection, SIEM use cases)
- Incident-response plan covering cloud-specific scenarios (account compromise, key compromise, cross-tenant)
- BCP and DR plan with tested recovery objectives
- Sub-processor register with evidence of oversight
- Customer notification procedure (incident, change, disruption)

## Required artefacts (CST)

- Cloud-use policy
- Tenant-side IAM with MFA on root / global admins
- Configuration baselines for the CSP services in use
- Tenant-side logging exported to entity-controlled SIEM
- Tenant-side encryption / key-management arrangements where the tier requires it
- Cloud incident-response runbooks
- Exit plan and portability test
- Risk-acceptance decisions for residual CSP-side risks
