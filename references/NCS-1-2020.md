## Purpose and Scope

NCS-1:2020 prescribes the **minimum acceptable cryptographic requirements** for national entities
when using cryptography to protect data (in use, at rest, and in transit), systems, and networks
for civilian or commercial purposes.

Covers: cryptographic primitives, schemes, prevalent protocols, Public Key Infrastructure (PKI),
and Key Lifecycle Management (KLM). Appendices address PRNG, Post-Quantum Cryptography,
and Side-Channel Attacks.

> **Compliance advisor note:** NCS-1:2020 is the cryptographic baseline underpinning all NCA
> controls frameworks. ECC-2:2024 §2-8 (Cryptography), DCC-1:2022, CSCC-1:2019, OTCC-1:2022,
> and CCC-2:2024 all reference NCS compliance. Other NCA regulations may mandate a **specific
> NCS strength level** (MODERATE or ADVANCED) for particular data classes or systems.
> Implementation vulnerabilities are explicitly flagged as a risk — correct algorithm selection
> alone does not guarantee secure implementation.

---

## Strength Levels

| Level | Security Target | Use |
|-------|----------------|-----|
| MODERATE | 128-bit security | Default minimum for civilian/commercial use |
| ADVANCED | 256-bit security | Higher-sensitivity data, systems, and networks |

Each national entity must **choose and implement the appropriate level** based on the nature and
sensitivity of the data, systems, and networks to be protected. Requirements not specifically
associated with one level apply equally to both.

## Section 2 — Cryptographic Primitives

---

### 2.1 Symmetric Algorithms

#### 2.1.1 Stream Cipher Algorithms

**Accepted stream ciphers:**

| Algorithm | MODERATE | ADVANCED | Standard |
|-----------|----------|----------|---------|
| SNOW 2.0 | 128-bit key | 256-bit key | ISO/IEC 18033-4 |
| SOSEMANUK | 128-bit and 256-bit key | **Not accepted** | eSTREAM project |

**Common requirements (both levels):**
- Initialization Vector (IV) must be **at least 128 bits**.
- Stream ciphers must be used with a **different IV for each key**.
- A key must be **used only once**.
- Valid decryption must **never be relied on for authenticity**.

> 🔴 **CRITICAL:** IV reuse with a stream cipher destroys confidentiality — keystream XOR reveals
> plaintext. IV uniqueness per key is non-negotiable.

---

#### 2.1.2 Block Cipher Algorithms

**Accepted block ciphers:**

| Algorithm | MODERATE key lengths | ADVANCED key length | Standard |
|-----------|---------------------|---------------------|---------|
| AES | 128-bit, 192-bit | 256-bit | FIPS-197 |
| Camellia | 128-bit, 192-bit | 256-bit | ISO/IEC 18033-3 |
| Serpent | 128-bit, 192-bit | 256-bit | FSE'98 |

> 🟠 **HIGH:** AES-128 is MODERATE; AES-256 is required for ADVANCED systems. Using AES-128
> where ADVANCED level is mandated by another NCA regulation is non-compliance.

---

### 2.2 Asymmetric Algorithms

**Accepted asymmetric algorithms:**

| Algorithm | MODERATE | ADVANCED | Notes |
|-----------|----------|----------|-------|
| RSA | ≥3072-bit key; *e* > 65537 | **Not accepted** | Strong primes must be used |
| Diffie-Hellman (Finite Field) | ≥3072-bit key; *q* subgroup size = 256 | **Not accepted** | — |
| ECDLP (NIST P-256, P-384, BrainpoolP256r1, BrainpoolP384r1, Curve25519) | ✅ | **Not accepted** | — |
| ECDLP (NIST P-521, Curve448, BrainpoolP512r1) | **Not accepted** | ✅ | Curve448 accepted despite 224-bit security due to side-channel resistance |

> 🔴 **CRITICAL:** RSA and Diffie-Hellman (finite field) are **not accepted at ADVANCED level**.
> Any system mandated to use ADVANCED that relies on RSA is non-compliant.
>
> 🟠 **HIGH — Strong primes:** RSA must use strong primes (*p*−1 has large prime factor *q*;
> *q*−1 has large prime factor; *p*+1 has large prime factor). Standard random prime selection
> without this property does not satisfy NCS.

---

### 2.3 Hash Functions

**Accepted hash functions:**

| Algorithm | MODERATE | ADVANCED | Notes |
|-----------|----------|----------|-------|
| SHA2-384 | ✅ | **Not accepted** | Also referred to as SHA-384 |
| SHA2-512/256 | ✅ | **Not accepted** | Also referred to as SHA-512/256 |
| SHA3-256 | ✅ | **Not accepted** | — |
| SHA3-384 | ✅ | **Not accepted** | — |
| SHAKE128 | ✅ (output *d* ≥256 bits) | **Not accepted** | — |
| SHAKE256 | ✅ (output *d* ≥512 bits) | **Not accepted** | — |
| SHA3-512 | **Not accepted** | ✅ | — |

**Common requirements:**
- Hash functions must be **inversion-resistant, pre-image resistant, and collision resistant**.
- SHAKE128: output size *d* ≥ 256 bits.
- SHAKE256: output size *d* ≥ 512 bits.

> 🟠 **HIGH:** SHA-1, MD5, SHA-256 alone, and SHA2-256 are **not in the NCS accepted list**.
> Systems using these are non-compliant. SHA2-384 is the minimum accepted SHA-2 variant.

---

### 2.4 Lightweight Crypto Algorithms

Accepted only on **limited systems with constrained resources where conventional cryptographic
standards are not applicable.**

**Accepted lightweight algorithms:**

| Type | Algorithm | Key / Output Size | Standard |
|------|-----------|------------------|---------|
| Block cipher | PRESENT | 80-bit or 128-bit key | ISO/IEC 29192-2 |
| Block cipher | CLEFIA | 128-bit, 192-bit, or 256-bit key | ISO/IEC 29192-2 |
| Stream cipher | Enocoro | 80-bit or 128-bit key | ISO/IEC 29192-3 |
| Stream cipher | Trivium | 80-bit key | ISO/IEC 29192-3 |
| Asymmetric | Unilateral auth mechanisms | — | ISO/IEC 29192-4 |
| Asymmetric | ALIKE key exchange | — | ISO/IEC 29192-4 |
| Asymmetric | Identity-based signature | — | ISO/IEC 29192-4 |
| Hash | PHOTON | 80/128/160/224/256-bit output | ISO/IEC 29192-5 |
| Hash | SPONGENT | 88/128/160/224/256-bit output | ISO/IEC 29192-5 |
| Hash | Lesamnta-LW | 256-bit output | ISO/IEC 29192-5 |
| MAC | Tsudik's keymode | Hash-based | ISO/IEC 29192-6 |
| MAC | Chaskey12 | 128-bit key | ISO/IEC 29192-6 |

> 🟡 **MEDIUM:** Lightweight algorithms are only acceptable where standard cryptography is
> not applicable. Using lightweight algorithms on systems capable of running standard AES is
> a non-compliance finding. OT/IoT environments using OTCC-1:2022 should document the
> justification for lightweight algorithm selection.

## Section 3 — Cryptographic Schemes

---

### 3.1 Block Cipher Modes of Operation

**Accepted modes:**

| Mode | Level | Standard |
|------|-------|---------|
| Counter Mode (CTR) | Both | NIST SP 800-38A |
| Cipher Block Chaining (CBC) | **MODERATE only** | NIST SP 800-38A |
| XEX Tweakable Block Cipher with Ciphertext Stealing (XTS-AES) | Both | NIST SP 800-38E |
| Output Feedback (OFB) | Both | NIST SP 800-38A |
| Cipher Feedback (CFB) | Both | NIST SP 800-38A |

> 🟠 **HIGH:** CBC is **only accepted for MODERATE**. Using CBC in an ADVANCED-level system is
> non-compliance. ECB mode is not listed and is not accepted.

---

### 3.2 Message Authentication Codes (MAC)

**Accepted MACs:**

| MAC | Key requirements | Tag length | Additional constraints | Standard |
|-----|-----------------|-----------|----------------------|---------|
| CMAC (Cipher-based MAC) | Protected authentication keys | ≥96 bits | Used for at most 2^48 messages; only in applications where no party learns all-0 string enciphering | NIST SP 800-38B |
| HMAC (Hash-based MAC) | Protected authentication keys | ≥96 bits | Used with SHA-2 and SHA-3 per §2.3 | FIPS PUB 198-1 |

> 🟠 **HIGH:** MAC tag length below 96 bits is non-compliance. CMAC usage limit of 2^48 messages
> must be tracked and enforced in implementation.

---

### 3.3 Authenticated Encryption with Associated Data (AEAD)

**Accepted AEAD schemes:**

| Scheme | Requirements | Standard |
|--------|-------------|---------|
| GCM (Galois Counter Mode) | Nonce ≥96 bits; IV unique within key change period; **short authentication tags not accepted** | NIST SP 800-38D |
| CCM (Counter with CBC-MAC) | — | NIST SP 800-38C |

> 🔴 **CRITICAL:** Short authentication tags in GCM are explicitly **not accepted**. Implementations
> using GCM with truncated tags (e.g., GCM-32 or GCM-64) are non-compliant. IV/nonce uniqueness
> per key is mandatory — IV reuse in GCM is catastrophic (forgery attack).

---

### 3.4 Key Wrap Functions

**Accepted key wrap functions:**
- Key Wrap (KW) — NIST SP 800-38F
- Key Wrap with Padding (KWP) — NIST SP 800-38F

---

### 3.5 Key Derivation Functions (KDF)

**Accepted KDFs:**
- RFC 5869 (HKDF)
- IKE-v2-KDF
- TLS-v1.2-KDF
- X9.63-KDF

**KDFs must comply with one of:**
- NIST 800-56 A/B KDF (Single Step)
- NIST 800-56 C KDF (Extract-then-expand)
- NIST 800-108

---

### 3.6 Key Agreement and Key Transport

#### 3.6.1 Symmetric Schemes
- Key agreement: must rely solely on **shared long-term secrets**.
- Key transport: all symmetric encryption schemes/primitives per §2 and §3; or combination
  of encryption scheme with MAC in **encrypt-then-MAC mode**.

#### 3.6.2 Asymmetric Key Exchange Algorithms

| Algorithm | MODERATE | ADVANCED | Notes | Standard |
|-----------|----------|----------|-------|---------|
| Diffie-Hellman (DH) | ≥3072-bit key | **Not accepted** | — | RFC 3526 |
| ECDH | 256–384-bit key | 512-bit key or Curve448 | Must implement forward secrecy and authenticated key establishment | NIST SP 800-56A |
| RSA Key Establishment | ≥3072-bit key | **Not accepted** | — | NIST 800-56B |

> 🔴 **CRITICAL:** ECDH without forward secrecy is non-compliant — the standard explicitly
> requires forward secrecy implementation.

---

### 3.7 Hybrid Encryption Schemes

| Scheme | MODERATE | ADVANCED |
|--------|----------|----------|
| ECIES (Elliptic Curve Integrated Encryption Scheme) | ✅ | ✅ |
| DLIES (Discrete Logarithm Integrated Encryption Scheme) | ✅ | **Not accepted** |
| RSA-OAEP (PKCS #1 v2.1) | ✅ | **Not accepted** |

---

### 3.8 Public Key Signatures

| Algorithm | MODERATE | ADVANCED | Standard |
|-----------|----------|----------|---------|
| DSA | ≥3072-bit modulus | **Not accepted** | FIPS PUB 186-4 |
| ECDSA | 256–384-bit key | 512-bit key or Curve448 | FIPS PUB 186-4 |
| RSA-PSS | 3072-bit modulus | **Not accepted** | PKCS RSA Cryptographic Standard v2.2 |
| RSA-DS2 (RSA Digital Signature Scheme) | 3072-bit modulus | **Not accepted** | ISO/IEC 9796-2-2010 |
| Merkle signatures | Accepted hash functions per §2 | Uses HMAC-based PRF on hash | — |

> 🟠 **HIGH:** DSA and RSA-PSS are MODERATE-only. ADVANCED systems must use ECDSA with ≥512-bit
> key or Curve448. Merkle signature PRF must be constructed with HMAC based on the hash function
> used — a custom PRF construction without this does not comply.

## Section 4 — Commonly Used Cryptographic Protocols

> Protocols not listed must follow the accepted primitives and schemes in §2 and §3.
> Future versions of listed protocols are accepted if they follow §2 and §3 primitives.

---

### 4.1 IP Security (IPsec)

**Authentication (AH and ESP):**

| MAC | Level |
|-----|-------|
| HMAC-SHA2-384, HMAC-SHA3-256, HMAC-SHA3-384 | MODERATE |
| HMAC-SHA3-512 | ADVANCED |

**Confidentiality (ESP with above MAC + one of):**
- AES-CTR
- CAMELLIA-CTR

**Alternative authenticated encryption modes:**

| Mode | ICV size | Level |
|------|---------|-------|
| AES-CCM_* | 12 or 16 bytes | Both |
| CAMELLIA-CCM_* | 12 or 16 bytes | Both |
| AES-GCM_* | 12 or 16 bytes | Both |

> 🟠 **HIGH:** ICV sizes below 12 bytes in IPsec AEAD modes are non-compliant. HMAC-SHA1 and
> HMAC-MD5 (common legacy IPsec defaults) are not in the accepted list.

---

### 4.2 Transport Layer Security (TLS)

**Accepted TLS versions:**
- **TLS 1.2** — acceptable using strongest cipher suites compatible with §2/§3; configuration
  must **not allow downgrading**.
- **TLS 1.3** — recommended for future applications.
- TLS versions below 1.2 are not accepted.

**Accepted cipher suites for TLS 1.2:**

| Cipher Suite | Level |
|-------------|-------|
| TLS_*1_*2_WITH_AES_128_CCM | MODERATE |
| TLS_*1_*2_WITH_AES_128_CCM_8 | MODERATE |
| TLS_*3_*4_WITH_Camellia_256_GCM_SHA-384 | MODERATE |
| TLS_*3_*4_WITH_AES_256_GCM_SHA-384 | MODERATE |
| TLS_*3_*4_WITH_AES_256_CCM | ADVANCED |
| TLS_*3_*4_WITH_AES_256_CCM_8 | ADVANCED |

Where: *1 = key agreement (ECDH/ECDHE/DH/DHE); *2 = signature (EC_DSA/RSA/DSS);
*3 = key agreement (ECDH/ECDHE); *4 = signature (EC_DSA).

**Accepted cipher suite for TLS 1.3:**
- TLS_AES_256_GCM_SHA384 — both MODERATE and ADVANCED.

> 🔴 **CRITICAL:** TLS 1.0 and TLS 1.1 are not accepted. Downgrade negotiation must be
> disabled in configuration. Cipher suites using RC4, DES, 3DES, NULL, EXPORT, or MD5 are
> not accepted. An auditor will check TLS configuration for version restrictions and cipher
> suite allowlist enforcement.

---

### 4.3 DNSSEC

**Zone data signing:**
- ECDSA_P384_SHA384 — both MODERATE and ADVANCED (special case: ECC 512-bit not implemented
  in this protocol).

**Message authentication:**

| Algorithm | Level |
|-----------|-------|
| HMAC-SHA384 | MODERATE |
| HMAC-SHA512 | ADVANCED (special case: SHA3-512 not implemented in this protocol) |

---

### 4.4 Secure Shell (SSH)

- **Accepted SSH version:** SSH-2 only. SSH-1 is not accepted.

**Accepted SSH encryption and MAC algorithms for SSH-2:**

| Algorithm | Level |
|-----------|-------|
| AEAD_AES_128_GCM | MODERATE |
| AEAD_AES_256_GCM | ADVANCED |

---

### 4.5 Bluetooth

**Accepted versions:** 4.1 or higher (NIST SP 800-121r2).

**Requirements:**
- Use Security Mode 4, Level 4 (Authenticated link key using secure connection).
- Use encryption algorithm **AES-CCM**.
- Use secure connection feature with **P-256 elliptic curve** for link key generation.
- Use encryption **Mode 3** (all traffic is encrypted).
- For Bluetooth Low Energy: version **4.2 or higher**, Low Energy Security Mode 1 Level 4.
- Organisations must use the **strongest Bluetooth security mode available** for their devices.

> 🟠 **HIGH:** Bluetooth versions below 4.1 are not accepted. Classic Bluetooth pairing modes
> below Security Mode 4 Level 4 are non-compliant. Unencrypted Bluetooth traffic (Mode 1 or 2)
> is a critical finding for any system handling sensitive data.

---

### 4.6 UMTS / LTE / 5G

| Network | Required algorithms |
|---------|-------------------|
| UMTS | 128-UEA1 and 128-UIA1 |
| LTE | 128-EEA2 and 128-EIA2 |
| 5G | 128-NEA2 and 128-NIA2 |

- EIA0 and NIA0: only acceptable in exceptional case of unauthenticated emergency calls in
  limited service mode.
- Only §2/§3 primitives must be used.
- **Exception:** KASUMI and both ECIES profiles (including the elliptic curves in the profile)
  are accepted solely for use under 3GPP systems.

---

### 4.7 WPA (Wi-Fi Protected Access)

**Accepted version:** WPA3-Enterprise only.

> 🔴 **CRITICAL:** WPA2 and WPA (first generation) are not in the accepted list. WPA3-Enterprise
> must be applied once it is available. For systems where WPA3-Enterprise is not yet supported by
> hardware, this creates a documented exception requiring compensating controls.

---

### 4.8 Kerberos Protocol

**Accepted Kerberos encryption types:**

| Encryption type | Level |
|----------------|-------|
| CAMELLIA128-CTS-CMAC | MODERATE |
| AES256-CTS-HMAC-SHA384 | MODERATE |
| CAMELLIA256-CTS-CMAC | ADVANCED |

> 🟠 **HIGH:** AES128-CTS-HMAC-SHA1-96 (a common legacy Kerberos default) is not listed in the
> accepted NCS types. Any Kerberos deployment using SHA-1-based encryption types is non-compliant.

## Section 5 — Public Key Infrastructure (PKI)

---

### 5.1 Algorithms for Certificates

**Root CA certificates:**

| Algorithm | Key size |
|-----------|---------|
| RSA | ≥4096 bits |
| ECC | NIST P-384, NIST P-521, Curve448, BrainpoolP384r1, or BrainpoolP512r1 |

**Intermediate CA and End User certificates:**

| Algorithm | Key size |
|-----------|---------|
| RSA | ≥3072 bits |
| ECC | NIST P-256, P-384, P-521, Curve25519, Curve448, BrainpoolP256r1, BrainpoolP384r1, or BrainpoolP512r1 |

**Common requirements:**
- Certificates, CRLs, and ARLs must comply with **X.509 PKI Certificate (RFC 5280)**.
- **SHA-2 or SHA-3** families must be used (per §2.3).
- **Match the strength** of asymmetric key algorithms with the strength of the hash algorithm.

> 🔴 **CRITICAL:** Root CA certificates using RSA below 4096 bits are non-compliant (note: stricter
> than the 3072-bit general minimum). SHA-1 signed certificates are non-compliant regardless of
> key size.
>
> 🟠 **HIGH:** Algorithm strength mismatch (e.g., P-256 key with SHA3-512 hash, or RSA-4096 with
> SHA2-256) violates the matching requirement.

---

### 5.2 Validity of Certificates

| Certificate type | Maximum validity |
|-----------------|----------------|
| Root CA | At most **20 years** |
| Intermediate CA / Subordinate CA / Issuing CA | At most **10 years** |
| End user — MODERATE | At most **5 years** |
| End user — ADVANCED | At most **3 years** |

> 🟠 **HIGH:** Certificates exceeding these validity periods are non-compliant at issuance.
> Existing certificates issued before NCS adoption require a migration plan. End-user certificates
> with 5+ year validity on ADVANCED systems are a finding.

## Section 6 — Key Lifecycle Management (KLM)

---

### 6.1 Key Protection and Lifetime

| Module type | MODERATE private key max lifetime | ADVANCED private key max lifetime |
|-------------|----------------------------------|----------------------------------|
| Hardware cryptographic modules | 5 years (does not limit CA cert lifetime) | 3 years (does not limit CA cert lifetime) |
| Software cryptographic modules | 2 years | **Not accepted** |

> 🔴 **CRITICAL:** Software cryptographic modules are **not accepted for ADVANCED** key protection.
> Private keys at ADVANCED level must be stored in hardware cryptographic modules (HSM or equivalent).
>
> 🟠 **HIGH:** Private keys exceeding the applicable lifetime must be revoked and replaced.
> Key lifecycle must be tracked with documented expiry dates.

---

### 6.2 KLM Processes

**Table 1 KLM Processes Requirements (verbatim from source):**

| KLM Process | Requirements |
|-------------|-------------|
| Key Generation | Keys (secret and private) must not be vulnerable to prediction or bias. Weak keys must not be used. Private and public keys require prime number generation with extra mathematical properties. |
| Key Registration / Certification | Keys must be associated with their owner (user) with a certificate. Root certificates must be distributed to relying parties and signatories. A trusted CA must be used. |
| Key Distribution and Installation | Keys must be distributed to their users securely and must be under the users' control. Keys must be transported securely by protecting their confidentiality and authenticity. All copies of keys must be securely installed and stored. Public keys must be transported and authenticity (but not privacy) must be protected by using certificates. Private keys must be protected and authorized by the owner/third party or CA. |
| Key Use | Keys must be protected against unauthorized use during their lifetimes. Keys must be protected against misuse from the owners, with key storage on secure hardware and/or secure software (authorization checks). |
| Key Storage | Organisations must require secure backups of keys (for internal or law enforcement use) when encryption is supported. Keys used for non-repudiation must be under the sole control of the user. |
| Key Revocation / Validation | In distributed systems, special measures such as updated versions of the CRL and OCSP must be implemented to avoid relying on keys that have expired. Key validation must be done by checking the CRL or OCSP servers. |
| Key Archive | The archival process must be secured, and confidentiality must be ensured to preserve the secrecy of information encrypted with archived keys. Expired keys must be archived to keep old data accessible when encryption is supported. Archival systems must follow the retention periods as per relevant regulations. |
| Key Destruction | When key lifetime expires and there is no need for it to be archived or stored, it must be removed from hardware via a secure deletion process. Media storage systems storing keys must be sanitized before disposal using a process compliant with NIST SP 800-88r1 or NSA/CSS Storage Device Sanitization Manual. |
| Key Accounting | There must be accounting for all asymmetric keys. Use of asymmetric keys must be monitored. There must be accounting for key use. |

#### Evidence Required — KLM

| KLM Process | Evidence Auditors Will Request |
|-------------|-------------------------------|
| Key Generation | Key generation procedure; HSM or PRNG documentation; proof weak keys are excluded |
| Key Registration | CA certificate issuance records; certificate-to-owner binding documentation |
| Key Distribution | Key distribution procedure; secure transport records; installation verification |
| Key Use | Access control policy for key use; HSM authorization logs; key use monitoring records |
| Key Storage | Secure backup procedure; non-repudiation key sole-control documentation |
| Key Revocation | CRL/OCSP configuration; revocation process documentation; revocation records |
| Key Archive | Secure archive procedure; retention schedule aligned with regulations; access controls on archive |
| Key Destruction | Secure deletion procedure; sanitization certificates (NIST SP 800-88r1 compliant) |
| Key Accounting | Asymmetric key inventory; key use audit logs; accounting reconciliation records |

## Section 7 — Appendices

---

### 7.1 Pseudo Random Number Generation (PRNG)

PRNG is required in several aspects of cryptography, including creation of asymmetric key-pairs
and symmetric keys.

**Requirements:**
- Software library functions such as `random()` must **not** be used — these are based on weak
  Linear Congruential Generators (LCG) and are cryptographically insecure.
- **Dedicated cryptographic PRNG implementations** are required for cryptographic applications.
- PRNGs should be tested with packages such as the NIST randomness tests (NIST SP 800-22) and
  the Dieharder test package.
- Applicable standards: ISO 28640:2010 and NIST SP 800-90A.

> 🔴 **CRITICAL:** Use of `rand()`, `random()`, `Math.random()` or equivalent language library
> PRNG functions in cryptographic key generation is a critical vulnerability. Auditors check
> cryptographic implementations for PRNG quality.

---

### 7.2 Post-Quantum Cryptography

- Quantum computers with sufficient qubits would break RSA, DSA, and ECDSA.
- At time of NCS-1:2020 publication: quantum computers with enough qubits not yet commercially
  available, but the risk to confidentiality of encrypted data is acknowledged as significant.
- No international standards for Post-Quantum Cryptography were available at publication time;
  expected from international standardization bodies within approximately three years of 2020.
- Post-Quantum Cryptography will be addressed in upcoming versions of the NCS.

> **2026 update note:** NIST finalised the first set of Post-Quantum Cryptography standards in
> 2024 (FIPS 203/ML-KEM, FIPS 204/ML-DSA, FIPS 205/SLH-DSA). NCS will need to be updated
> to reference these. Organisations planning long-lived cryptographic infrastructure should begin
> crypto-agility planning now. [verify ref — check NCA for any NCS version update post-2020]

---

### 7.3 Side-Channel Attacks

Side-channel attacks exploit physical measurements (energy consumption, electromagnetic emanation,
time consumption) to gain insight into sensitive data. They can be carried out by remote and
passive adversaries and may go undetected.

**Mitigation requirements:**
- Cryptographic operations must be performed in **certified hardware components** (e.g., smart
  cards) to protect secret/private keys.
- Analyse effects of side channels on certified hardware in a **specialized laboratory** during
  development.
- **Protect ciphertexts using MAC and verify ciphertexts** before performing any other
  cryptographic operations.
- Eliminate correlation between binary representation of secret key and side-channel signals
  (e.g., use dummy operations to mask correlation).
- Side-channel attacks cover a wide range of scenarios — an **extensive review of relevant
  threat scenarios** should be carried out for proper secure implementation.

## Compliance Quick Reference — MODERATE vs ADVANCED

| Control area | MODERATE | ADVANCED |
|-------------|----------|----------|
| **Symmetric block cipher** | AES-128, AES-192, Camellia-128/192, Serpent-128/192 | AES-256, Camellia-256, Serpent-256 |
| **Asymmetric** | RSA ≥3072; DH ≥3072; ECDLP P-256/P-384/Curve25519 | ECDLP P-521/Curve448/BrainpoolP512r1 only |
| **Hash** | SHA2-384, SHA2-512/256, SHA3-256/384, SHAKE128/256 | SHA3-512 only |
| **Block cipher mode** | CTR, CBC, XTS-AES, OFB, CFB | CTR, XTS-AES, OFB, CFB (no CBC) |
| **TLS** | 1.2 with approved suites; 1.3 recommended | 1.2 with AES-256 suites; 1.3 recommended |
| **SSH** | AEAD_AES_128_GCM | AEAD_AES_256_GCM |
| **Kerberos** | CAMELLIA128-CTS-CMAC, AES256-CTS-HMAC-SHA384 | CAMELLIA256-CTS-CMAC |
| **Private key lifetime (HW)** | ≤5 years | ≤3 years |
| **Private key lifetime (SW)** | ≤2 years | **Not accepted** |
| **End user cert validity** | ≤5 years | ≤3 years |
| **Root CA cert validity** | ≤20 years | ≤20 years |
| **WPA** | WPA3-Enterprise | WPA3-Enterprise |

## Critical Audit Flags — NCS-1:2020

| Severity | Area | Finding |
|----------|------|---------|
| 🔴 CRITICAL | Stream ciphers | IV reuse with same key |
| 🔴 CRITICAL | GCM | Nonce reuse or short authentication tags |
| 🔴 CRITICAL | TLS | TLS 1.0/1.1 in use; downgrade negotiation enabled |
| 🔴 CRITICAL | WPA | WPA2 or WPA1 in use (WPA3-Enterprise not deployed) |
| 🔴 CRITICAL | PRNG | Language library `random()` used in key generation |
| 🔴 CRITICAL | Advanced level | RSA or DH (finite field) used at ADVANCED level |
| 🔴 CRITICAL | Advanced level | Software crypto module used for ADVANCED private keys |
| 🔴 CRITICAL | Root CA | RSA Root CA key below 4096 bits |
| 🔴 CRITICAL | ECDH | Forward secrecy not implemented |
| 🟠 HIGH | Block ciphers | AES-128 used where ADVANCED level mandated |
