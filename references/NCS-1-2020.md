## At a glance

| Field | Value |
|-------|-------|

| Relationship to other NCA frameworks | Referenced as the cryptographic baseline in ECC-2:2024 §2-5, DCC-1:2022, CSCC-1:2019, OTCC-1:2022, and other NCA controls frameworks — compliance with those controls requires NCS compliance |

## Purpose and scope

NCS-1:2020 prescribes the **minimum acceptable cryptographic requirements** for national entities when using cryptography to protect data (in use, at rest, and in transit), systems, and networks for civilian or commercial purposes.

The standard covers:
- Cryptographic primitives (symmetric and asymmetric algorithms, hash functions, lightweight algorithms)
- Cryptographic schemes (block cipher modes, MAC, AEAD, key wrap, KDF, key agreement/transport, hybrid encryption, public key signatures)
- Commonly used cryptographic protocols (IPsec, TLS, DNSSEC, SSH, Bluetooth, UMTS/LTE/5G, WPA, Kerberos)
- Public Key Infrastructure (PKI) — certificate algorithms and validity periods
- Key Lifecycle Management (KLM)

The NCS was built assuming **absence of quantum computing capabilities**. Post-quantum cryptography will be addressed in future versions.

## Two strength levels — the core NCS framework

The NCS defines two strength levels. Every entity selects the level appropriate to the sensitivity of the data, system, or network being protected. Other NCA regulations may mandate a specific level.

| Level | Target Security | Practical meaning |
|-------|----------------|------------------|
| **MODERATE** | 128-bit security | Standard government and commercial systems; required minimum for most ECC control contexts |
| **ADVANCED** | 256-bit security | Higher-sensitivity data and systems; Secret/Top Secret classification under DCC-1:2022; systems designated critical under CSCC-1:2019 |

> **Critical cross-framework note:** DCC-1:2022 §2-5-1 explicitly requires MODERATE for Confidential data and ADVANCED for Secret and Top Secret data. When advising on DCC-1:2022 compliance, always map the data classification to the required NCS level.

## §2 — Cryptographic Primitives

### §2.1.1 — Stream Cipher Algorithms

| Algorithm | MODERATE | ADVANCED | Notes |
|-----------|----------|----------|-------|
| SNOW 2.0 (ISO/IEC 18033-4) | 128-bit key | 256-bit key | IV ≥ 128 bits; unique IV per key; key used only once; decryption must not be relied on for authenticity |
| SOSEMANUK (eSTREAM project) | 128-bit or 256-bit key | **Not accepted** | IV ≥ 128 bits; same common notes apply |

Common notes (all stream ciphers): IV must be at least 128 bits; unique IV per key; each key used only once; valid decryption must never be relied on for authenticity.

### §2.1.2 — Block Cipher Algorithms

| Algorithm | MODERATE key lengths | ADVANCED key length |
|-----------|---------------------|---------------------|
| AES (FIPS-197) | 128-bit or 192-bit | 256-bit |
| Camellia (ISO/IEC 18033-3) | 128-bit or 192-bit | 256-bit |
| Serpent | 128-bit or 192-bit | 256-bit |

### §2.2 — Asymmetric Algorithms

| Algorithm | MODERATE | ADVANCED | Notes |
|-----------|----------|----------|-------|
| RSA | ≥ 3072-bit key; e > 65537 | **Not accepted** | Strong primes must be used |
| Diffie-Hellman and Finite Field | ≥ 3072-bit key; q (subgroup size) = 256 | **Not accepted** | — |
| ECDLP-based (ECC) | NIST P-256, P-384, BrainpoolP256r1, BrainpoolP384r1, Curve25519 | NIST P-521, Curve448, BrainpoolP512r1 | Curve448 accepted for ADVANCED despite 224-bit security due to performance and side-channel resistance |

### §2.3 — Hash Functions

| Algorithm | MODERATE | ADVANCED | Notes |
|-----------|----------|----------|-------|
| SHA-2 | SHA2-384, SHA2-512/256 | **Not accepted** | Also known as SHA-384, SHA-512/256 |
| SHA-3 | SHA3-256, SHA3-384, SHAKE128, SHAKE256 | SHA3-512 | SHAKE128: output ≥ 256 bits; SHAKE256: output ≥ 512 bits |

Common notes: hash functions must be inversion-resistant, pre-image resistant, and collision resistant.

### §2.4 — Lightweight Crypto Algorithms

For resource-constrained devices where conventional algorithms are not applicable (ISO/IEC 29192 series):

| Category | Accepted algorithms |
|----------|---------------------|
| Block ciphers (ISO/IEC 29192-2) | PRESENT (80-bit or 128-bit key); CLEFIA (128-bit, 192-bit, or 256-bit key) |
| Stream ciphers (ISO/IEC 29192-3) | Enocoro (80-bit or 128-bit key); Trivium (80-bit key) |
| Asymmetric mechanisms (ISO/IEC 29192-4) | Unilateral authentication mechanisms; ALIKE key exchange; Identity-based signature |
| Hash functions (ISO/IEC 29192-5) | PHOTON (80/128/160/224/256-bit output); SPONGENT (88/128/160/224/256-bit output); Lesamnta-LW (256-bit output) |
| MAC (ISO/IEC 29192-6) | Tsudik's keymode (hash-based); Chaskey12 (128-bit key) |

## §3 — Cryptographic Schemes

### §3.1 — Block Cipher Modes of Operation

| Mode | Accepted for | Standard |
|------|-------------|----------|
| CTR (Counter Mode) | MODERATE + ADVANCED | NIST SP 800-38A |
| CBC (Cipher Block Chaining) | **MODERATE only** | NIST SP 800-38A |
| XTS-AES (XEX Tweakable Block Cipher with Ciphertext Stealing) | MODERATE + ADVANCED | NIST SP 800-38E |
| OFB (Output Feedback) | MODERATE + ADVANCED | NIST SP 800-38A |
| CFB (Cipher Feedback) | MODERATE + ADVANCED | NIST SP 800-38A |

### §3.2 — Message Authentication Codes (MAC)

| MAC | Requirements |
|-----|-------------|
| CMAC (NIST SP 800-38B) | Used for at most 2⁴⁸ messages; not used where any party learns all-0 string enciphering; tag ≥ 96 bits with protected authentication keys |
| HMAC (FIPS PUB 198-1) | Tag ≥ 96 bits with protected authentication keys; used with SHA-2 and SHA-3 per §2.3 |

### §3.3 — AEAD (Authenticated Encryption with Associated Data)

| Scheme | Requirements |
|--------|-------------|
| GCM (NIST SP 800-38D) | Nonce ≥ 96 bits; IV unique within key change period; short authentication tags **not accepted** |
| CCM (NIST SP 800-38C) | Accepted |

### §3.4 — Key Wrap Functions

- Key Wrap (KW) as in NIST SP 800-38F
- Key Wrap with Padding (KWP) as in NIST SP 800-38F

### §3.5 — Key Derivation Functions (KDF)

Accepted KDF functions: RFC 5869 (HKDF); IKE-v2-KDF; TLS-v1.2-KDF; X9.63-KDF.

KDFs must comply with one of: NIST-800-56 A/B KDF (Single Step); NIST-800-56 C KDF (Extract-then-expand); NIST-800-108.

### §3.6 — Key Agreement and Key Transport

**Symmetric schemes:**
- Key agreement: must rely solely on shared long-term secrets.
- Key transport: all symmetric encryption schemes and primitives from §2 and §3 may be used; combination of encryption scheme with MAC in encrypt-then-MAC mode is accepted.

**Asymmetric key exchange (§3.6.2):**

| Algorithm | MODERATE | ADVANCED | Notes |
|-----------|----------|----------|-------|
| DH (RFC 3526) | ≥ 3072-bit key | **Not accepted** | — |
| ECDH (NIST SP 800-56A) | 256–384-bit key | 512-bit key or Curve448 | With forward secrecy and authenticated key establishment |
| RSA Key Establishment (NIST 800-56B) | ≥ 3072-bit key | **Not accepted** | — |

### §3.7 — Hybrid Encryption Schemes

| Scheme | MODERATE | ADVANCED |
|--------|----------|----------|
| ECIES (Elliptic Curve Integrated Encryption Scheme) | ✓ | ✓ |
| DLIES (Discrete Logarithm Integrated Encryption Scheme) | ✓ | **Not accepted** |
| RSA-OAEP (PKCS #1 v2.1) | ✓ | **Not accepted** |

### §3.8 — Public Key Signatures

| Scheme | MODERATE | ADVANCED | Notes |
|--------|----------|----------|-------|
| DSA (FIPS PUB 186-4) | ≥ 3072-bit modulus | **Not accepted** | — |
| ECDSA (FIPS PUB 186-4) | 256–384-bit key | 512-bit key or Curve448 | Auth algorithms dependent on system/use case |
| RSA-PSS / RSA-DS2 (PKCS v2.2) | 3072-bit modulus | **Not accepted** | Both RSA-PSS and RSA-DS2 accepted for MODERATE |
| Merkle signatures | Both levels | Both levels | Must use accepted hash functions from §2; pseudo-random function family constructed with HMAC based on the hash function used |

## §4 — Commonly Used Cryptographic Protocols

### §4.1 — IPsec

| Function | MODERATE | ADVANCED |
|----------|----------|----------|
| Authentication (AH/ESP MAC) | HMAC-SHA2-384, HMAC-SHA3-256, HMAC-SHA3-384 | HMAC-SHA3-512 |
| Confidentiality (ESP encryption) | AES-CTR, CAMELLIA-CTR | (same, combined with ADVANCED MAC) |
| Authenticated encryption (alternative) | AES-CCM_12, AES-CCM_16, CAMELLIA-CCM_12, CAMELLIA-CCM_16, AES-GCM_12, AES-GCM_16 | Same modes |

### §4.2 — TLS

- TLS 1.2 acceptable with strongest primitives and no-downgrade configuration; TLS 1.3 recommended for future applications.
- TLS 1.3 accepted cipher suite: **TLS_AES_256_GCM_SHA384** (both MODERATE and ADVANCED).

| TLS 1.2 cipher suites | Level |
|----------------------|-------|
| TLS_*1_*2_WITH_AES_128_CCM / AES_128_CCM_8, TLS_*3_*4_WITH_Camellia_256_GCM_SHA-384, TLS_*3_*4_WITH_AES_256_GCM_SHA-384 | MODERATE |
| TLS_*3_*4_WITH_AES_256_CCM, TLS_*3_*4_WITH_AES_256_CCM_8 | ADVANCED |

Where: *1 = key agreement (ECDH, ECDHE, DH, DHE); *2 = signature (EC_DSA, RSA, DSS); *3 = key agreement (ECDH, ECDHE); *4 = signature (EC_DSA).

### §4.3 — DNSSEC

| Function | MODERATE | ADVANCED |
|----------|----------|----------|
| Zone data signing | ECDSA_P384_SHA384 | ECDSA_P384_SHA384 (special case — ECC 512-bit not implemented in protocol) |
| Message authentication | HMAC-SHA384 | HMAC-SHA512 (special case — SHA3-512 not implemented in protocol) |

### §4.4 — SSH

- Accepted version: SSH-2 only.
- MODERATE: AEAD_AES_128_GCM
- ADVANCED: AEAD_AES_256_GCM

### §4.5 — Bluetooth

- Accepted versions: 4.1 or higher (NIST SP 800-121r2).
- Requirements: Security Mode 4 Level 4; AES-CCM encryption; P-256 elliptic curve for link key generation; Encryption Mode 3 (all traffic encrypted).
- Low Energy: version 4.2 or higher, Low Energy Security Mode 1 Level 4.
- Organisations must use the strongest Bluetooth security mode available for their devices.

### §4.6 — UMTS / LTE / 5G

| Standard | Required algorithms |
|----------|---------------------|
| UMTS | 128-UEA1 and 128-UIA1 |
| LTE | 128-EEA2 and 128-EIA2 |
| 5G | 128-NEA2 and 128-NIA2 |
| Exception | EIA0 and NIA0 only for unauthenticated emergency calls in limited service mode |
| 3GPP exception | KASUMI and both ECIES profiles (including their elliptic curves) accepted solely for use under 3GPP systems |

### §4.7 — WPA (Wi-Fi Protected Access)

- **WPA3-Enterprise only.** Must be applied once available; no other WPA version is accepted.

### §4.8 — Kerberos

| Level | Accepted encryption types |
|-------|--------------------------|
| MODERATE | CAMELLIA128-CTS-CMAC, AES256-CTS-HMAC-SHA384 |
| ADVANCED | CAMELLIA256-CTS-CMAC |

## §5 — Public Key Infrastructure (PKI)

### §5.1 — Algorithms for Certificates

| Certificate type | RSA key length | ECC curves |
|-----------------|---------------|------------|
| Root CA | ≥ 4096 bits | NIST P-384, P-521, Curve448, BrainpoolP384r1, BrainpoolP512r1 |
| Intermediate CA / End User | ≥ 3072 bits | NIST P-256, P-384, P-521, Curve25519, Curve448, BrainpoolP256r1, BrainpoolP384r1, BrainpoolP512r1 |

Common notes: certificates, CRLs, and ARLs must comply with X.509 PKI Certificate (RFC 5280); SHA-2 or SHA-3 must be used; asymmetric key algorithm strength must match hash algorithm strength.

### §5.2 — Validity of Certificates

| Certificate type | Maximum validity |
|-----------------|-----------------|
| Root CA | 20 years |
| Intermediate CA / Subordinate CA / Issuing CA | 10 years |
| End user — MODERATE | 5 years |
| End user — ADVANCED | 3 years |

## §6 — Key Lifecycle Management (KLM)

### §6.1 — Key Protection and Lifetime

| Module type | MODERATE private key lifetime | ADVANCED private key lifetime |
|-------------|------------------------------|------------------------------|
| Hardware cryptographic module | ≤ 5 years (does not limit CA certificate lifetime) | ≤ 3 years (does not limit CA certificate lifetime) |
| Software cryptographic module | ≤ 2 years | **Not accepted** |

### §6.2 — KLM Processes (Table 1)

| KLM Process | Key Requirements |
|-------------|-----------------|
| Key Generation | Keys (secret and private) must not be vulnerable to prediction or bias; weak keys must not be used; private and public keys require prime number generation with extra mathematical properties |
| Key Registration / Certification | Keys must be associated with their owner (user) with a certificate; root certificates distributed to relying parties and signatories; trusted CA must be used |
| Key Distribution and Installation | Distributed securely under user control; transported with confidentiality and authenticity protection; all copies securely installed and stored; public keys transported with authenticity (not privacy) protected via certificates; private keys protected and authorised by owner/third party/CA |
| Key Use | Protected against unauthorised use during lifetime; protected against misuse by owners, with key storage on secure hardware and/or secure software (authorisation checks) |
| Key Storage | Secure backups of keys required (internal or law enforcement use) when encryption is supported; keys used for non-repudiation must be under sole control of the user |
| Key Revocation / Validation | CRL and OCSP must be updated in distributed systems; key validation must be done by checking CRL or OCSP servers |
| Key Archive | Archival process secured and confidentiality ensured; expired keys archived to keep old data accessible when encryption is supported; archival retention periods must comply with relevant regulations |
| Key Destruction | Removed from hardware via secure deletion process when no longer needed; media must be sanitised per NIST SP 800-88r1 or NSA/CSS Storage Device Sanitization Manual |
| Key Accounting | Accounting for all asymmetric keys; use of asymmetric keys monitored; accounting for key use |

## §7 — Appendices: advisory considerations

### §7.1 — Pseudo Random Number Generation (PRNG)

- Software library functions such as `random()` must **not** be used — based on weak Linear Congruential Generators (LCG).
- Dedicated cryptographic PRNG implementations are required.
- PRNGs should be tested with NIST SP 800-22 randomness tests and/or Dieharder test package.
- Reference standards: ISO 28640:2010 and NIST SP 800-90A.

### §7.2 — Post-Quantum Cryptography

- RSA, DSA, and ECDSA are expected to be broken by quantum computing once sufficient qubits are available.
- At time of NCS-1:2020 publication, no international standards for post-quantum cryptography existed; expected to be issued by international bodies within approximately three years.
- Post-quantum cryptography to be addressed in future NCS versions.

> **Advisor note (May 2026):** NIST finalised its first post-quantum cryptography standards in August 2024 (FIPS 203 ML-KEM, FIPS 204 ML-DSA, FIPS 205 SLH-DSA). NCS has not yet been updated to incorporate these. Entities with ADVANCED-level obligations or long-lived key infrastructure should monitor NCA for an NCS update.

### §7.3 — Side-Channel Attacks

Side-channel attacks exploit physical measurements (power consumption, electromagnetic emanation, timing) to extract key material. Requirements to minimise risk:
- Cryptographic operations must be performed in certified hardware components (e.g., smart cards).
- Side-channel effects on hardware components must be analysed in a specialised laboratory during development.
- Ciphertexts must be protected using MAC and verified before performing further cryptographic operations.

## Common pitfalls

1. **Assuming ECC compliance implies NCS compliance.** ECC-2:2024 §2-5 requires NCS compliance for cryptography, but ECC does not re-enumerate algorithm specifics. Entities that have passed an ECC audit but never performed a dedicated NCS review frequently have unchecked algorithm gaps (legacy RSA-2048, SHA-1, TLS 1.1).

2. **Not linking data classification to NCS level.** The MODERATE/ADVANCED selection is not arbitrary. DCC-1:2022 §2-5-1 mandates ADVANCED for Secret and Top Secret data. Advisors must map the entity's data classification scheme to NCS levels and verify every system handling classified data uses the correct level.

3. **Treating RSA as future-safe for ADVANCED.** RSA is accepted only for MODERATE — it is explicitly not accepted for ADVANCED. Any system requiring ADVANCED protection must use ECDLP-based algorithms (NIST P-521, Curve448, BrainpoolP512r1).

4. **Overlooking the software key storage prohibition for ADVANCED.** Software cryptographic modules are not accepted for ADVANCED private key storage. This catches many cloud-native deployments that store private keys in software-based secrets managers rather than cloud HSM services.

5. **Certificate validity misconfigured relative to level.** A common gap: end-user certificates valid for 5 years are compliant for MODERATE but non-compliant for ADVANCED (maximum 3 years). Teams that apply a single certificate template across all systems risk non-compliance on ADVANCED systems.

6. **Post-quantum gap not tracked.** NCS-1:2020 acknowledges that RSA, DSA, and ECDSA are quantum-vulnerable. NIST published its first post-quantum standards in August 2024. Entities with long-lived ADVANCED infrastructure should already have a post-quantum migration roadmap. Absence of any planning is a governance gap under ECC-2:2024 §1-5 (Risk Management).

7. **Lightweight cryptography used without justification.** §2.4 algorithms are accepted only where conventional standards are not applicable (resource-constrained devices). Using lightweight algorithms on capable systems without documented justification is a compliance gap.

8. **GCM nonce reuse.** NCS §3.3 requires the GCM IV to be unique within the key change period. Nonce reuse in GCM breaks confidentiality and authenticity catastrophically. This is a common implementation error in developer-built encryption rather than library-managed encryption.

## Key definitions (from NCS §7.4)

| Term | NCS Definition |
|------|---------------|
| Asymmetric Algorithm | A cryptographic algorithm using one key for encryption and another for decryption; the two keys are called private and public keys |
| Authentication | Verifying the identity of a user, process, or device, often as a prerequisite to allowing access to resources |
| Authenticity | The property of being genuine and being able to be verified and trusted |
| Block Cipher Algorithm | A symmetric key cipher method that segments data into blocks and encrypts each one separately |
| Certificate | A set of data uniquely identifying an entity's public key and other information, digitally signed by a CA, binding the public key to the owner |
| CRL (Certificate Revocation List) | A list of revoked certificates issued by a Certification Authority |
| CA (Certification Authority) | A trusted entity responsible for issuing and revoking public key certificates |
| Collision | Two or more distinct inputs producing the same output |
| Confidentiality | A property preventing information from being available or disclosed to unauthorised individuals, entities, or processes |
| Cryptographic Primitive | A low-level cryptographic algorithm used as a basic building block for higher-level algorithms |
| Cryptography | Principles, means, and methods of applying data transformation algorithms for security purposes including integrity, confidentiality, authentication, authenticity, and non-repudiation |
| ECC | Public-key cryptographic methods using operations in an elliptic curve group |
| Encryption | The process of transforming plaintext into ciphertext using a cryptographic algorithm and key |
| Hash Function | A function mapping an input bit string of arbitrary length to a fixed-length output; often irreversible |
| HMAC | A message authentication code using an approved keyed-hash function |
| Hybrid encryption | Cryptography combining two or more encryption algorithms, particularly symmetric and asymmetric |
| Initialization Vector (IV) | A known public vector used as input to initialise an encryption algorithm to increase security and support synchronisation |
| Integrity | A property whereby data has not been altered in an unauthorised manner since creation, transmission, or storage |
| KLM (Key Lifecycle Management) | Activities involving handling of cryptographic keys and related security parameters across their entire lifecycle, from generation through destruction |
| Key Agreement | A key-establishment procedure where keying material is a function of contributions from two or more participants |
| Key Derivation Functions | The process by which one or more keys are derived from a pre-shared key or shared secret |
| Key Destruction | Removing all traces of keying material so it cannot be recovered by physical or electronic means |
| Key Wrap | A method of encrypting keys (with integrity information) providing both confidentiality and integrity protection using a symmetric key algorithm |
| Lightweight Crypto | A sub-category of cryptography providing security solutions for resource-constrained devices |
| MAC | A cryptographic checksum on data using a symmetric key to detect accidental and intentional modifications; provides authenticity and integrity protection |
| Non-repudiation | A service using a digital signature supporting determination of whether a message was actually signed by a given entity |
| PKI | A framework established to issue, maintain, and revoke public key certificates |
| RSA | An asymmetric algorithm used for key establishment and digital signature generation and verification |
| Stream Cipher Algorithm | A symmetric key cipher where each plaintext bit/word is encrypted one at a time with a corresponding pseudorandom keystream |
| Strong Primes | Primes whose products are hard to factor; prime p is strong if: (p−1) has a large prime factor q, (q−1) has a large prime factor, and (p+1) has a large prime factor |
| Symmetric Algorithm | A cryptographic algorithm using a single secret key for both encryption and decryption |
