## Key definitions (§2)

| Term | Definition |
|------|-----------|
| Cloud Computing | Use of a pool of flexible, scalable and sharable physical and virtual resources (servers, OS, networks, programs, applications, storage) plus automatic on-demand service management |
| Cloud Computing Services / Computing Services | Providing ICT services through cloud computing, including storing, transmitting or processing subscriber content in a cloud computing system. Three service types: SaaS, PaaS, IaaS |
| SaaS (§2-1-2-1) | Provider-hosted applications accessed by subscriber via thin client (browser); subscriber does not manage underlying infrastructure |
| PaaS (§2-1-2-2) | Platform for subscriber to deploy and run applications developed or purchased by the subscriber; subscriber controls published applications and possibly hosting environment configurations |
| IaaS (§2-1-2-3) | Primary computing resources (processing, storage, networking) provided to subscriber; subscriber controls OS, storage and published applications, and may have limited control over some network components |
| Cloud Computing Service Provider (CSP) | Any person providing cloud computing services directly or indirectly: includes Service Provider (§2-1-3-1), Cloud Computing Service Broker (§2-1-3-2), and Cloud Computing Service Aggregator (§2-1-3-3) |
| Service Provider (§2-1-3-1) | Person providing cloud computing services through data centers **owned and managed partially or completely by that same person** |
| Cloud Computing Service Broker (§2-1-3-2) | Intermediary between one or more CSPs and subscribers |
| Cloud Computing Service Aggregator (§2-1-3-3) | Broker that aggregates and combines multiple services into combined packages |
| Cloud Computing Subscriber | Person to whom the CSP provides services under a cloud computing contract or other commercial relationship |
| Cloud Computing User | Natural person who uses the services provided to the subscriber; may simultaneously be the subscriber |
| Cloud Computing Nodes (§2-1-6) | Any agreement between a CSP and its subscribers to provide services |
| Cloud Computing System (§2-1-7) | Electronic information system comprising hardware, software and network elements owned, controlled, operated, leased or otherwise used by the CSP; may include one or more data centers |
| Public Cloud Computing (§2-1-8) | Cloud computing system available for open use by any entity or individual |
| Community Cloud (§2-1-9) | Cloud computing system for exclusive use of a closed group sharing social, business, administrative or other goals |
| Private Cloud Computing (§2-1-10) | Cloud computing system for exclusive use of one subscriber |
| Hybrid Cloud Computing System (§2-1-11) | Combination of two or more public, private and/or community cloud systems linked together |
| Data Center (§2-1-12) | Facility containing essential cloud computing infrastructure and supporting components, used at least partially for storing and/or processing subscriber content and data |
| Content (§2-1-13) | Any programs, texts, files, sound recordings, images, animations, captions, information, personal or commercial data, or any other data in any form |
| Subscriber Content (§2-1-14) | Any content provided or produced by the subscriber that is saved or processed in the cloud computing system pursuant to the cloud computing contract |
| Subscriber Data (§2-1-15) | Data falling under at least one of three categories: (1) personal identification data of cloud computing users; (2) business, financial affairs or commercial data of the subscriber; (3) data produced by the CSP relating to the subscriber's activity log, billing, usage volume, statistics |
| Subscriber Address (§2-1-16) | Contract address or billing address; if both exist and only one is in the Kingdom, the Kingdom address is the Subscriber Address |
| Third-Party Content (§2-1-17) | Content from anyone other than the CSP or subscriber, provided to the subscriber through or in conjunction with their use of the services |
| Law-Violating Content (§2-1-18) | Subscriber or third-party content that violates the Kingdom's laws |
| IPR-Violating Content (§2-1-19) | Subscriber or third-party content that infringes intellectual property rights |
| Residence (§2-1-20) | Permanent or temporary residence per the Kingdom's laws; does not include accommodation for short visits or transit |
| Service Credits (§2-1-21) | Compensation mechanisms (discounts, free service extensions) provided to subscribers when provider performance fails to meet SLA standards or regulatory requirements |
| Service Level Agreement (§2-1-22) | Agreement between CSP and subscriber defining the quality of services in measurable standards |

## §3-1 Scope

| Provision | §-ref | Detail |
|-----------|-------|--------|
| Primary territorial scope | §3-1-1 | Applies to cloud computing services provided to subscribers residing in or having a Subscriber Address in the Kingdom |
| Data-centre-based scope (regardless of subscriber location) | §3-1-2 | Processing or storing content/data of any subscriber (temporarily or permanently) in data centres or cloud computing system elements **located in the Kingdom** triggers: (a) §3-5-4 and §3-5-5 (content removal obligations); (b) §3-5-6 (Anti-Cyber Crime Law notification); (c) §3-4-3-1 exception; and (d) registration requirements §3-2 |
| Obligation binding on contracting CSP | §3-1-3 | Obligations under §3-1-1 bind the CSP that entered into the cloud computing contract with the subscriber |
| Provisions are mandatory | §3-1-4 | Unless otherwise specified in these regulations, provisions are mandatory and cannot be amended during the validity of the contract |

## §3-2 Registration to Provide Cloud Computing Services

| Provision | §-ref | Detail |
|-----------|-------|--------|
| Registration required before operating in-Kingdom infrastructure | §3-2-1 | No CSP may exercise direct or effective control over a data centre or critical infrastructure of a cloud computing system hosted and used in the Kingdom (in whole or in part) for providing cloud computing services **before completing correct and complete registration with CST**; communications infrastructure including international communications must use CST-authorised channels |
| Registration requirements are in the Guide | §3-2-2 | Registration requirements and procedures are in the "Guide for Cloud Computing Service Providers in the Kingdom of Saudi Arabia" published on CST's website (Arabic approved version is authoritative) |
| Qualifying category | §3-2-3 | CST may register providers in a qualifying category to overcome challenges of categorisation under Table 2; CST specifies requirements and procedures |

### Registration categories (Table 2 — verified from source)

| Registration Category | Government agency data handling capability |
|----------------------|-------------------------------------------|
| (a) | Public data only |
| (B) | Public + Restricted data |
| (C)² | Public + Restricted + Secret + Top Secret |

> **Note 1 (from source):** The registration category determines the subscriber data classification the CSP is qualified to handle, per Tables 1 and 2. The "Guide for Cloud Computing Service Providers in the Kingdom" explains special requirements for each category.

> **Note 2 (from source):** To handle government agency data classified as Secret or Top Secret, it requires referring to CST to obtain necessary approvals in accordance with applicable laws, regulations, policies, and governance models in the Kingdom.

## §3-3 Subscriber Data

### Subscriber data classification (§3-3-1 — Table 1)

Subscriber data may be classified at different levels depending on NDMO guidance or the regulatory authorities supervising the subscriber:

**Data of Saudi Government Agencies:**

| Level | Trigger for classification |
|-------|---------------------------|
| Top Secret (Extremely Confidential) | Unauthorised access or disclosure causes serious and exceptional damage that cannot be remedied to: national interests (breaching agreements/treaties, harming reputation, diplomatic relations, political affiliations, or operational efficiency of security/military operations, national economy, national infrastructure, government business); performance of government agencies harmful to national interest; broader individual health and safety and privacy of senior officials; environmental or natural resources |
| Secret (Confidential) | Unauthorised access or disclosure causes serious and exceptional damage to: national interests (partially harming reputation, diplomatic relations, political affiliations, or operational efficiency of security/military/economy/infrastructure/government business); causes financial loss at organisational level leading to bankruptcy, inability to perform duties, serious loss of competitiveness; causes serious harm affecting life of a group of individuals; leads to long-term environmental damage; investigating major cases including terrorism financing |
| Restricted | Unauthorised access leads to: limited negative impact on government agency work or economic activities or a specific person; limited damage to entity assets and financial/competitive position; limited short-term environmental damage |
| Public | Unauthorised access does not result in any of the above effects on: national interest; entity activities; interests of individuals; environmental resources |

**Data of Non-Government Agencies:**
- Data received from Saudi Government Agencies: classified as received from the government agency based on the levels above
- Other data: not covered by the above; NDMO and sector-regulator guidance applies

### Subscriber data classification responsibility (§3-3-4 to §3-3-5)

| Obligation | §-ref |
|-----------|-------|
| Subscribers shall choose the appropriate classification from Table 1 or an equivalent CSP-provided list conforming to their specific needs, obligations and duties | §3-3-4 |
| Subscribers whose data is classified as Data of Saudi Government Agencies must contract with a CST-registered service provider | §3-3-4 |
| CSP must notify its subscribers in writing or electronically of its CST registration category, including the classification level it can handle per Tables 1 and 2 | §3-3-5 |

### Subscriber content site and transfer — data localisation (§3-3-6 to §3-3-8)

| Obligation | §-ref |
|-----------|-------|
| CST-registered CSPs and subscribers must **not transfer any content from Saudi government agencies' data outside the Kingdom** for any purpose, in any form, whether permanently or temporarily (including temporary storage and backup), unless expressly permitted by the laws or regulations in the Kingdom | §3-3-6 |
| Subscribers may not transfer, store or process shared content from Saudi government agencies' data to any **public, community or hybrid cloud system** belonging to a provider in the Kingdom, unless the CSP is properly registered with CST | §3-3-7 |
| CST-registered CSPs must **clearly notify subscribers in advance** and obtain their consent **before** transferring, storing or processing their content outside the Kingdom permanently or temporarily | §3-3-8 |

### Reporting data breaches (§3-3-9 to §3-3-10)

| Obligation | §-ref |
|-----------|-------|
| CSP must notify **CST and its subscribers without undue delay** of any breaches of user information or documents that it becomes aware of; CST will notify NDMO if breaches affect or are likely to affect government agencies or a large number of people in the Kingdom | §3-3-9 |
| CSP must notify subscribers of any insurance coverage provided by the CSP against civil liability; information must include at minimum the basic characteristics reasonably required for the subscriber to assess their risk exposure | §3-3-10 |

## §3-4 Subscriber Data Protection

| Obligation | §-ref |
|-----------|-------|
| Applies to CSPs who: (a) enter into a cloud computing contract; and (b) those who specify goals and means of processing subscriber data even without being a direct contract party | §3-4-1 |
| CSP may **not** provide, authorise or enable any third party to access or use content or data related to the subscriber | §3-4-2-1 |
| CSP may **not** process or use subscriber content or data for purposes other than those permitted under the cloud computing contract; must disclose (in a designated detailed document) any capabilities to view, decode or assist third parties in accessing data stored, processed or transferred in/through the Kingdom; must obtain CST's **explicit and prior written approval** before implementing any new such capabilities | §3-4-2-2 |
| Exceptions to §3-4-2 (do not apply when): (a) subscriber requires the CSP to disclose/transmit/process/use data in accordance with the laws of the Kingdom; or (b) data is non-governmental and the subscriber gives prior explicit consent (with right to withdraw at any time) | §3-4-3 |
| CSP must grant subscribers the **technical right and ability to access, verify, correct and delete their data**, in a manner that does not contradict NDMO guidance on personal data | §3-4-4 |
| Exception: §3-4-4 obligations do not impair the CSP's rights over activity log, billing and usage data (§2-1-15-3) to the extent necessary for billing purposes or compliance with Kingdom laws | §3-4-5 |
| §3-4 provisions apply without prejudice to any statutory, regulatory or contractual provisions in the Kingdom providing a higher degree of protection for personal or commercial data | §3-4-6 |

## §3-5 Law-Violating Content and IPR-Violating Content

| Obligation | §-ref |
|-----------|-------|
| CSP is not liable merely because subscriber content violates the law or infringes IPR, so long as the CSP did not have actual knowledge and has not been notified by a competent authority | §3-5-2 |
| **No general monitoring obligation**: these regulations do not create an obligation on CSPs to proactively monitor their cloud systems to identify law-violating or IPR-violating content | §3-5-3 |
| **Mandatory removal on CST/authority direction**: if CST or any competent authority directs the CSP **in writing** to remove law-violating or IPR-violating content from any data centre or cloud system element in the Kingdom, the CSP must ensure the content is **removed** or **made inaccessible in the Kingdom** | §3-5-4 |
| CSP may proactively (on own initiative or at third-party request) remove or make inaccessible IPR-violating content, provided it is in accordance with the cloud computing contract and the affected subscriber is given sufficient notice | §3-5-5 |
| CSP must notify **CST and/or any competent authority without undue delay** if it discovers any content or information in the cloud computing system that may be violating the laws and legislation of the Kingdom (cybersecurity matters excepted — those go via NCA channels) | §3-5-6 |
| Third-party IPR complaints must be referred to competent authorities unless the CSP decides to address directly under §3-5-5 | §3-5-7 |
| CSP has the right to notify the subscriber that their content has been found violating or has been removed, unless CST or another competent authority prevents this (and such prevention requires relevant justification) | §3-5-8 |
| §3-5 provisions do not prejudice the CSP's obligation to cooperate with competent authorities under applicable laws and registration conditions | §3-5-9 |
| CSP must grant subscribers all necessary statutory licences to use software or other IP protected works provided under the cloud computing contract, proportionate to term and scope | §3-5-10 |

## §3-6 Cloud Computing Service Contracts — Mandatory Minimum Content

### Pre-contract disclosure (§3-6-1)

Before signing the contract, the CSP must provide clear and transparent information to the subscriber about: the subject of the service; terms of use; service levels; payment mechanism; and the CST registration category (including the subscriber data classification level per Tables 1 and 2).

### Mandatory contract content (§3-6-3)

Cloud computing service contracts must include **at minimum**:

| Minimum contract element | §-ref |
|--------------------------|-------|
| CSP information: business address and full contact details | §3-6-3-1 |
| Statement of services to be provided and their permitted uses | §3-6-3-2 |
| Contract term (if applicable), applicable fees, payment terms, and termination provisions | §3-6-3-3 |
| Rules for handling subscriber content, including processing and the process for the subscriber to obtain a copy upon contract termination | §3-6-3-4 |
| Information regarding availability, terms and conditions of any SLA that the CSP can make available | §3-6-3-5 |
| Procedures for resolving subscriber complaints | §3-6-3-6 |
| The applicable judicial law for interpretation of the contract or dispute resolution — provided it cannot nullify any provision of these regulations or other mandatory KSA laws that may not be nullified by choice of international law | §3-6-3-7 |

### Additional contract obligations (§3-6-4 to §3-6-6)

| Obligation | §-ref |
|-----------|-------|
| CSP must provide a customer care service to resolve subscriber complaints; this does not prejudice any other legal remedies under applicable law | §3-6-4 |
| CSPs and subscribers have the right to refer disputes to CST's dispute resolution procedures, without prejudice to other available procedures | §3-6-5 |
| Upon termination of the cloud computing contract, the CSP must enable the subscriber to obtain a copy of their cloud computing content for a certain period of time and in the specific means and format in accordance with the contract | §3-6-6 |

## §3-7 Subscriber Protection and Unfair Contract Terms

| Provision | §-ref |
|-----------|-------|
| CSP bears responsibility before CST and its individual subscribers for any acts or negligence by itself, its agents, subcontractors or employees acting within the framework of their engagement | §3-7-1 |
| CSP cannot disclaim contractual liability for: (a) loss or damage to subscriber content or data related to CSP processing or handling; (b) quality, performance, accessibility, duration of downtime, or other service standards that do not match the contract or mandatory statutory provisions | §3-7-2 |
| A "best endeavours" clause does not exempt the CSP from liability for intentional acts or gross negligence to individual subscribers | §3-7-3 |

## CSP registration categories — practical decision guide

| Entity situation | Registration requirement |
|-----------------|-------------------------|
| Operating data centres partially or wholly in KSA for cloud services | Must register with CST before operating (§3-2-1) |
| Providing services only to subscribers with KSA address, but no KSA data centres | Scope applies per §3-1-1; registration requirement under §3-1-2-3 if using KSA infrastructure |
| Handling Saudi government agency Public data only | Category (a) registration |
| Handling Saudi government agency Restricted data | Category (B) registration |
| Handling Saudi government agency Secret or Top Secret data | Category (C)² registration — requires additional CST approval |
| Acting as broker/aggregator | Must still meet registration requirements applicable to the category of data being handled |
| Foreign CSP with no KSA data centres, serving KSA subscribers | §3-1-1 scope applies; §3-1-2 scope applies if any subscriber content is stored/processed in KSA infrastructure even temporarily |

## Audit hot spots — ranked by enforcement frequency

| Rank | Issue | §-ref | Why it gets cited |
|------|-------|-------|-------------------|
| 1 | Operating KSA-hosted cloud infrastructure without CST registration or in a higher category than registered | §3-2-1, §3-3-4 | Gateway obligation; CST can suspend/cancel registration immediately |
| 2 | Government agency data transferred or backed up outside the Kingdom without express legal permission | §3-3-6 | Absolute prohibition for Saudi government data; frequently violated by international cloud providers using global backup infrastructure |
| 3 | Subscriber not notified before out-of-Kingdom transfer, storage or processing | §3-3-8 | Advance notice + consent required; often absent in standard CSP ToS for cross-border DR/backup |
| 4 | Cloud contract missing mandatory minimum content (one or more of the 7 required elements) | §3-6-3 | CST inspects contract templates; missing SLA terms and data-return procedures are common gaps |
| 5 | No pre-contract disclosure of CST registration category and subscriber data classification capability | §3-6-1, §3-3-5 | Subscriber cannot make an informed choice of classification without this; frequently missing for resellers and brokers |
| 6 | CSP disclosing subscriber data to third parties without authority direction or subscriber consent | §3-4-2-1 | Explicit prohibition; enterprise data shared with analytics or AI training platforms is the current risk area |
| 7 | Processing subscriber data beyond the purposes in the cloud computing contract | §3-4-2-2 | Use-limitation obligation; CSP secondary analytics use cases are a common breach |
| 8 | New data-access or decryption capability deployed without prior CST written approval | §3-4-2-2 | Must obtain explicit and prior written CST approval — frequently not obtained |
| 9 | Breach notification to CST and subscribers not made without undue delay | §3-3-9 | "Without undue delay" standard; no specific hour count defined — CST interprets this strictly |
| 10 | No mechanism for subscribers to access, verify, correct and delete their data | §3-4-4 | Technical right and ability must be provided; subscriber rights are frequently not surfaced in the service interface |
| 11 | SLA performance data not accessible to subscribers for last 12 months | §3-8-1-1 | CST checks this at registration and at audit; absent for many reseller arrangements |
| 12 | No business continuity management system documented or implemented | §3-11-3, §3-8-2 | BCM summary must be provided to subscribers; risk register required; frequently treated as internal document |
| 13 | Government agency data stored/processed on un-registered public, community or hybrid cloud in KSA | §3-3-7 | Subscribers (government entities) bear this obligation; unregistered providers routinely used |
| 14 | Foreign law chosen in contract to nullify mandatory KSA provisions | §3-6-3-7 | Governing law clauses pointing to DIFC, English or US law cannot override mandatory CST/NCA/NDMO obligations |

## Common pitfalls

- **Registration category mismatch:** A CSP registered in Category (B) accepting government-classified Secret data is in material breach. Category determines what data may be handled — not all CSPs check the subscriber's data classification before onboarding.
- **Global backup and DR for government data:** The §3-3-6 prohibition on transferring government data outside the Kingdom applies to temporary storage and backup explicitly. Global cloud providers running standard geo-redundant architectures are routinely in breach for government customers.
- **Broker/aggregator responsibility:** Service brokers and aggregators are CSPs under the definitions (§2-1-3-2, §2-1-3-3). They bear the same obligations as direct service providers. Resellers frequently believe the underlying provider's registration covers them — it does not.
- **New decryption/access capability without CST approval:** §3-4-2-2 requires CST's explicit and prior written approval before implementing any new capability to view, decode or access subscriber data. AI-driven data analytics features added to cloud platforms routinely trigger this requirement and are launched without it.
- **Foreign governing law clauses:** Many multinational CSPs use English, DIFC or US-law governed contracts. §3-6-3-7 requires that the chosen governing law cannot nullify mandatory KSA regulatory provisions. A governing-law clause that purports to do this is itself void but creates enforcement ambiguity.
- **"Without undue delay" breach notification — no clock defined:** Unlike PDPL's 72-hour standard, CST CCSR §3-3-9 says "without undue delay" with no specific hour threshold. CST interprets this strictly. CSPs treating this as a 5-business-day window face enforcement risk.
- **Subscriber data rights vs billing data carve-out:** §3-4-4 grants subscribers the right to access, verify, correct and delete their data. §3-4-5 carves out activity logs, billing and usage data for billing and legal compliance purposes. CSPs often extend the carve-out to all operational data — the carve-out is narrow.
- **Content filtering exclusion is CST's decision:** The exclusion from content filtering under §3-9-1 (for private cloud or subscriber-controlled networks) requires a CST **decision** — it is not automatic. Assuming the exclusion applies without CST confirmation is a compliance error.
